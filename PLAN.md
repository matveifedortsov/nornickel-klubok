# План доработок «Научный клубок» до полного соответствия ТЗ

## Контекст

Текущий скелет (`klubok/`) реализует архитектуру «граф+вектор+Mock/real» на **демо-домене
термообработки сплавов** (Cu-Ni, отжиг, твёрдость по Виккерсу). Официальное ТЗ трека шире и
конкретнее: **R&D-карта горно-металлургической отрасли** (гидро-/пирометаллургия, обогащение,
экология, геомеханика) с требованиями, которых в скелете нет — верификация фактов, гео-фильтр
РФ/мир, числовые диапазоны в запросах, версионирование, RBAC+аудит, экспорт, дашборды,
сравнительный анализ. Загружен реальный корпус (1453 файла, 4.9 ГБ: журналы, конференции,
внутренние статьи/обзоры/доклады) и подтверждено, что **GPU и MetalGPT-1 будут доступны** до
сдачи — значит план ориентируется на реальный прогон через MetalGPT-1 + BGE-m3, а не только Mock.

План исчерпывающий (без урезания под дедлайн, по решению пользователя), разбит на независимые
воркстримы в порядке зависимостей: сначала контракт данных (онтология), потом извлечение и
ингест реального корпуса, потом поиск/аналитика, потом API/UI/безопасность, потом упаковка сдачи.

## 1. Онтология и модель данных — `klubok/ontology.py`

Домен-специфичная схема (сплавы/отжиг) заменяется/расширяется под ТЗ:

- **Новые типы узлов** (`NodeType`): `Equipment`, `Facility` (лаборатория/установка),
  `Expert` (автор/носитель компетенции), `Publication` (заменяет узкий `Document` — статья/отчёт/
  патент/норматив), `Condition` (набор условий эксперимента: климат, географию, режим).
  Сохранить `Material, Process, Property, Experiment, Method, Phase`, обобщить их описания под
  металлургию (не только сплавы: сульфаты/хлориды/гипс/угольные отходы, кучное выщелачивание,
  электроэкстракция, обессоливание воды и т.д.) — обновить докстроки и, где были узко
  материаловедческие assumptions (напр. `resolver.canonical_material` заточен под сплавы),
  явно оставить как fallback-ветку, не удаляя.
- **Новые типы связей** (`RelType`): `USES_MATERIAL`, `OPERATES_AT_CONDITION`,
  `PRODUCES_OUTPUT`, `DESCRIBED_IN` (Experiment/Process → Publication), `VALIDATED_BY`
  (Fact → Expert/Publication), `CONTRADICTS` (Relation/Experiment ↔ Relation/Experiment),
  `AUTHORED_BY` (Publication → Expert), `EXPERT_IN` (Expert → Process/Material — область
  экспертизы), `AFFILIATED_WITH` (Expert → Facility). Существующие связи (`USES`, `APPLIES`,
  `MEASURES`, `RESULTS_IN`, `REPORTS`→переименовать семантику на Publication, `EXHIBITS`,
  `OBSERVED_BY`, `CITES`) остаются, `ALLOWED_EDGES` расширяется новыми парами.
- **Верификация фактов**: добавить в `Relation` поля `source_type` (публикация/внутренний
  отчёт/эксперимент), `verification_level` (подтверждено/предварительно/спорно — enum),
  `actualized_at` (дата, ISO-строка) — рядом с уже существующими `evidence`, `confidence`.
- **Гео-признак**: поле `geography` (строка страна/регион) + производный булев `is_domestic`
  на `Publication`/`Experiment`/`Relation` — фильтр «отечественная vs мировая практика».
- **Числовые ограничения как структура, не строка**: новый pydantic-тип `NumericConstraint`
  (`param: str, operator: Literal["<=",">=","=","between"], value: float, value_high: float|None,
  unit: str`) — используется в `Entity.attributes` для Property/Process/Condition и в
  распознавании запросов пользователя (см. §4). Это прямое требование ТЗ («сульфаты ≤300 мг/л»,
  «производительность от 100 т/сут»).
- **Домен/тег** (`domain: str` на Publication/Experiment — «гидрометаллургия», «пирометаллургия»,
  «экология», «переработка отходов», «геомеханика» …) — свободный тег, не enum, чтобы
  расширяемость (нефункц. требование) не упиралась в код.

Обновить `klubok/graph/schema.cypher`: constraints на новые labels (`Equipment`, `Facility`,
`Expert`, `Publication`), fulltext index расширить на новые типы + добавить range-индексы на
числовые атрибуты, которые часто фильтруются (`CREATE INDEX ... FOR ()-[r:...]-() ON (r.value)`
где применимо), и индекс на `geography`/`domain` для быстрой фильтрации.

**Версионирование фактов**: вместо мутации ребра при повторном ингесте — при конфликте
(новое значение при том же `(src,rel,dst)`, но другое `value`/`evidence`) не перезаписывать
`MERGE...SET`, а создавать новую версию ребра с `valid_from`/`superseded_by`, старую помечать
`is_current=false`. Меняет `klubok/graph/ingest.py::ingest_extraction` — вместо безусловного
`SET rel.evidence=...` сначала `MATCH` существующее ребро, сравнить `evidence`/`value`, при
расхождении создать доп. ребро-версию с относящей связью `SUPERSEDES` либо доп. atributami
`versions: [...]` (проще для Cypher — выбрать второе: параллельные помеченные рёбра одного типа
с разным `valid_from`, ретривал фильтрует `is_current=true` по умолчанию).

## 2. Извлечение под реальный домен — `klubok/extraction/`

- **`prompts.py`**: заменить/дополнить `FEW_SHOT_EXAMPLES` парой реалистичных металлургических
  примеров на русском (напр. циркуляция католита при электроэкстракции никеля; обессоливание
  воды обогатительной фабрики с сульфатами/хлоридами/Ca/Mg/Na) — few-shot должен матчить домен
  реального корпуса, иначе MetalGPT будет извлекать в demo-стиле сплавов. Обновить `_RULES` под
  новые типы узлов/связей из §1, добавить инструкцию извлекать `NumericConstraint` (оператор +
  диапазон) и `geography`/`actualized_at`, если явно упомянуты в тексте.
- **`normalize.py`**: расширить `_UNIT_ALIASES`/`_DIMENSION` под металлургические единицы —
  концентрация (мг/л, мг/дм³, г/л), расход/производительность (м³/ч, т/сут, л/мин), скорость
  потока, электрические (А/м², В). Добавить парсинг операторов сравнения и диапазонов —
  новая функция `parse_constraint(text) -> NumericConstraint | None`, распознающая «≤300 мг/л»,
  «от 100 т/сут», «200–300 мг/л» (regex на `≤|<=|не более|до`, `≥|>=|от|не менее`, тире-диапазон).
  Это прямое требование ТЗ («корректное распознавание числовых ограничений — ошибки недопустимы»)
  и основа для структурного (не только векторного) поиска в §4.
- **`resolver.py`**: словарь синонимов (`ALIAS_MAP`) расширить RU/EN парами из ТЗ
  («электроэкстракция»/«electrowinning», «ПВП»/«печь взвешенной плавки»/«fluidized bed
  furnace», «выщелачивание»/«leaching», «флотация»/«flotation» и т.п.) — вынести в отдельный
  файл-глоссарий `klubok/extraction/glossary_ru_en.py` (или CSV/YAML под `klubok/data/`), чтобы
  расширять его без правки кода — попадает под нефункц. требование «расширяемость» и «FAIR».
- **`extractor.py`**: `parse_extraction` — добавить парсинг новых полей связи (`source_type`,
  `verification_level`, `actualized_at`, `geography`) из JSON-ответа LLM с безопасными
  дефолтами (если LLM не вернул — `verification_level="unverified"`, `geography=None`).
- **Парсинг не-PDF форматов**: корпус на 115 docx + 46 xls + 5 pptx — `klubok/parsing/pdf_parser.py`
  сейчас берёт только PDF. Добавить `klubok/parsing/docx_parser.py` (python-docx, ленивый импорт
  как в pdf_parser) и `klubok/parsing/pptx_parser.py` (python-pptx, текст по слайдам) с тем же
  контрактом `-> Document`, переиспользуя `chunk_text` из pdf_parser.py (не дублировать). Обновить
  `pipeline.ingest_path` — определять парсер по расширению файла, а не хардкодить `*.pdf`.
- **Метаданные из имени файла** (папка «Статьи»): автор и лаборатория закодированы в имени
  (`26 Статья - Великая Т.И. (ИАЦ).docx`) — добавить `klubok/parsing/filename_meta.py` с regex-
  извлечением `author_name`, `lab_abbr` без LLM; в `pipeline.ingest_document` создавать
  `Expert`/`Facility` узлы и `AUTHORED_BY`/`AFFILIATED_WITH` рёбра напрямую, до/параллельно с
  LLM-извлечением — дешёвый и точный сигнал, LLM на нём может ошибаться.

## 3. Ингест реального корпуса — `klubok/pipeline.py`, новый `scripts/ingest_corpus.py`

- **Отбор корпуса**: не гнать весь 4.9 ГБ через LLM. Новый скрипт `scripts/select_corpus.py`,
  строящий список файлов для ингеста: включает «Статьи» (60, полностью — там же лежит авторство),
  «Обзоры» (104), по 2 последних номера каждого журнала, исключает `Материалы конференций/
  Источники данных о *` (рыночная аналитика CRU/Brook Hunt/ICSG — не про технологии) и все
  zip/rar (не распакованы, вне контракта парсера).
- **`scripts/ingest_corpus.py`**: батч-ингест с логированием прогресса, ретраями на ошибки LLM/
  парсинга одного файла (не должен ронять весь батч — обязательное требование «надёжность» из
  ТЗ), опциональной конкурентностью (`asyncio`/`ThreadPoolExecutor`, т.к. MetalGPT — сеть) и
  чекпоинтом (файл прогресса, чтобы прервать/продолжить длинный прогон). Пишет сводку
  entities/relations/ошибок в конце — переиспользует `pipeline.ingest_document`, ничего не
  дублирует.
- **Надёжность парсинга** — обернуть `parse_pdf`/`docx_parser`/`pptx_parser` вызовы в try/except
  с логом файла-виновника и продолжением батча (сканы без текстового слоя, битые файлы — в таком
  корпусе гарантированно есть).

## 4. Гибридный поиск с числовыми/гео-фильтрами — `klubok/retrieval/graphrag.py`

- **Разбор вопроса на структурные ограничения**: новая функция `extract_query_constraints(question)`
  в `graphrag.py`, переиспользующая `normalize.parse_constraint` (§2) — вытаскивает из текста
  вопроса числовые условия («сухой остаток ≤1000 мг/дм³») и гео-маркеры («в России», «мировая
  практика» → `geography`/`is_domestic`).
- **Структурный Cypher-фильтр параллельно вектору**: `expand_subgraph` дополняется опциональными
  `WHERE`-условиями по `r.value`/`r.unit`/`n.geography`/`n.domain`, собранными из
  `extract_query_constraints`. Это отдельная функция `filtered_expand(client, seeds, constraints)`,
  не ломающая текущий безусловный путь.
- **Глубина обхода 3-4 хопа + производительность**: текущий `_EXPAND_QUERY` — 1 хоп. Заменить на
  variable-length path (`-[r*1..4]->`) с `LIMIT` и, для приемлемой скорости на объёме до 1 млн
  сущностей (требование «3-5 сек»), обязательно завести индексы на часто фильтруемые свойства
  (`geography`, `domain`, числовые атрибуты — см. §1 schema.cypher) и ограничить fan-out на
  каждом хопе (`LIMIT` внутри `apoc.path.subgraphAll` вместо чистого `MATCH`, т.к. APOC уже
  подключен в `docker-compose.yml`).
- **`RetrievalContext`**: добавить поле `constraints: list[NumericConstraint]` и
  `geography_filter: str | None` для прозрачности в UI/ответе (жюри должно видеть, что фильтр
  применился, не только смысловой поиск).

## 5. Аналитика: пробелы, версии, сравнение — `klubok/graph/gaps.py`, новый `klubok/analytics/`

- **Расширить `gaps.py`** запросами из ТЗ, которых нет: `only_domestic_or_foreign(client)` —
  технологии, описанные только в отечественных ИЛИ только в зарубежных источниках (по
  `geography`/`is_domestic`); `unstudied_combinations(client, material, condition)` — обобщение
  текущего `MATERIAL_PROCESS_NO_EFFECT` на произвольную комбинацию материал-режим-условие;
  `contradictions(client)` — рёбра/эксперименты, связанные `CONTRADICTS`, с обоими источниками
  для ручного разбора аналитиком.
- **Новый модуль `klubok/analytics/compare.py`**: сравнительные таблицы «технология А vs Б» по
  параметрам (эффективность/CAPEX/применимость в холодном климате/экологические ограничения) —
  Cypher-запрос, собирающий `Property`/`Condition` по двум подграфам Process/Material и
  агрегирующий в таблицу строк-параметров × столбцов-вариантов. Отдельная функция, не встроена в
  `gaps.py`, т.к. другой характер (сравнение, не поиск дыр).
- **Новый модуль `klubok/analytics/recommend.py`**: «похожие кейсы» — переиспользует
  `ANALOGY_GAPS`-паттерн (общий Element/Material → разница в измеренных Property) обобщённо на
  Process/Condition; «эксперты по теме» — обход `Expert -[:EXPERT_IN]-> Process/Material`,
  ранжирование по числу `AUTHORED_BY` публикаций по теме.
- **Дашборд-метрики**: `klubok/analytics/dashboard.py` — агрегаты по `domain` (покрытие числом
  Publication/Experiment на домен), по командам/Facility (активность = число публикаций за период),
  «зоны риска» = темы с числом источников ниже порога (переиспользует `sparse_properties`-паттерн,
  обобщённый на любой тип узла через параметр).

## 6. Ответы: провенанс, обзорный синтез — `klubok/qa/answer.py`

- **Формат цитаты с верификацией**: `build_answer_prompt`/`ANSWER_INSTRUCTION` — добавить в
  контекст `verification_level`/`actualized_at`/`geography` из `graph_context_text()` (сейчас
  туда идёт только `evidence`+`doc_id`), чтобы LLM явно писал «подтверждено N источниками,
  уровень достоверности — высокий/предварительный» — прямое требование ТЗ по аналитике.
- **Режим «литературный обзор»**: новая функция `generate_literature_review(topic, client, store,
  llm)` в `qa/answer.py` (или новый `klubok/qa/review.py`, если разрастётся) — не просто
  question-answer, а структурированный синтез: группировка найденных `Publication` по
  методу/году/географии, отдельная секция «консенсус» vs «разногласия» (используя `CONTRADICTS`
  из §1/§5), степень уверенности = доля источников с высоким `verification_level`. Переиспользует
  `retrieve()` из graphrag.py с более широким `top_k`, но собирает `Answer`-подобную структуру
  `LiteratureReview` с секциями.

## 7. API — `klubok/api/app.py`

- **Новые эндпоинты**: `POST /compare` (тела: два набора material/process/condition →
  compare.py), `GET /experts?topic=` и `GET /facilities` (обход Expert/Facility), `POST /review`
  (литературный обзор из §6), `GET /dashboard` (метрики из §5), `GET /export?format=pdf|md|jsonld&
  answer_id=` (см. ниже).
- **Фильтры на существующих**: `/ask` и `/subgraph` принимают опциональные `geography`,
  `domain`, `date_from`, `date_to` — пробрасываются в `retrieve()` (§4).
- **Экспорт**: новый `klubok/export/` — `to_markdown(answer)`, `to_json_ld(answer)` (простая
  сериализация под JSON-LD с `@context` минимальной схемой Publication/Experiment/Material),
  `to_pdf(answer)` (переиспользовать `anthropic-skills:pdf`-подход недоступен в рантайме сервиса —
  использовать `reportlab`/`weasyprint` на markdown→pdf, т.к. это должно работать без интернета
  на бэкенде). Формат ответов уже структурирован (`Answer`/`LiteratureReview`), экспорт — чистое
  форматирование поверх них, никакой новой бизнес-логики.
- **RBAC + аудит**:
  - Ролевая модель: `researcher, analyst, project_lead, admin, external_partner` — простая
    реализация без внешнего IdP (нет времени/инфры на полноценный auth-провайдер): API-key в
    заголовке → роль из конфигурируемой мапы (`config.py: settings.api_keys: dict[str, str]`,
    `.env`). Дальше — FastAPI `Depends` на роль, минимально: `external_partner` не видит
    `Publication` с `sensitivity=internal` (новое поле на Publication из §1) и не имеет доступа к
    `/gaps`, `/dashboard`; остальные роли — полный доступ, `admin` дополнительно может дергать
    ручную корректировку графа (§9).
  - Аудит: middleware в `app.py`, логирующий `(timestamp, api_key/role, method, path, params)` в
    файл или отдельную Neo4j-ноду `AuditLog` (проще — файл/SQLite, граф не место для логов) —
    минимальная реализация, не over-engineering.
- **Ручная корректировка графа** (эксперт правит связи): `POST /graph/edge` (создать/обновить
  ребро вручную, `admin`/`project_lead` роль), `PATCH /graph/edge/{id}/comment` — комментарий +
  автор/дата на ребро (`edited_by`, `edited_at`, `comment` — новые поля ребра). Переиспользует
  `_merge_entity`/MERGE-паттерн из `graph/ingest.py`, не новая логика записи.

## 8. UI (Streamlit) — `ui/streamlit_app.py`

- **Визуализация подграфа графом** (не список текста) — `streamlit-agraph` или `pyvis` рендер
  `sg["edges"]`/`sg["seeds"]` из уже существующего `/subgraph` — главное визуальное усиление,
  явно упомянутое в ТЗ («отображение цепочек материал→процесс→оборудование→результат»,
  «подсветка противоречивых данных/пробелов»).
- **Панель фильтров** над вкладкой «Вопрос»: гео (РФ/мир/все), домен (мультиселект), диапазон
  дат, порог confidence — прокидываются в `/ask` и `/subgraph` (§7).
- **Вкладка «Сравнение»**: форма выбора двух технологий/материалов → таблица из `/compare`.
- **Вкладка «Дашборд»** (для роли `project_lead`/`admin`): метрики из `/dashboard` — покрытие по
  доменам, активность команд, зоны риска — карточки+график (`st.bar_chart`).
- **Вкладка «Эксперты»**: поиск экспертов/лабораторий по теме (`/experts`).
- **Экспорт-кнопки** на вкладке «Вопрос»: скачать ответ как PDF/MD/JSON-LD (`/export`).
- **Роль пользователя**: простой selectbox «Роль» в сайдбаре, хранящий выбранный API-key/роль в
  `st.session_state`, подставляется в заголовки запросов к API (соответствует RBAC из §7; полноценный
  логин не нужен для демо-жюри, но модель должна быть видна).
- **Мультиязычность вопроса**: т.к. BGE-m3 и MetalGPT-1 многоязычны by design, отдельного UI-кода
  не требуется — важно только не терять кириллицу/латиницу в `chunk_text`/regex (уже ОК, но
  добавить тестовый EN-вопрос в `eval/gold_set.py`, см. §9).

## Порядок реализации (зависимости, не дедлайн)

1. §1 Онтология + schema.cypher (фундамент, всё остальное на неё опирается).
2. §2 Извлечение (промпты/normalize/resolver/парсеры docx-pptx/filename_meta) — без этого нечем
   наполнять новый граф.
3. §3 Ингест реального корпуса — первый реальный прогон на MetalGPT-1, даёт данные для всего
   нижестоящего.
4. §4 Retrieval с фильтрами, §5 Аналитика, §6 QA-ответы — параллельно, все читают уже
   наполненный граф.
5. §7 API, §8 UI — тонкие слои поверх §4-6, делать последними по каждому эндпоинту/вкладке.
6. §9 Тесты/eval — сопровождают каждый шаг, не отдельная финальная фаза.

## Верификация

- `pytest -q` после каждого воркстрима (юнит-тесты чистой логики — normalize/resolver/parsers
  не требуют GPU/БД).
- `python scripts/eval_extraction.py` — численные precision/recall после смены промптов (§2),
  сравнение до/после на gold-наборе.
- `docker compose up -d` + реальный `scripts/ingest_corpus.py` на отобранном сабсете (§3) —
  ручная проверка через Neo4j Browser (`http://localhost:7474`), что новые типы узлов/связей и
  верификационные поля появились.
- End-to-end через UI (`streamlit run ui/streamlit_app.py`): прогнать 4 эталонных вопроса из ТЗ
  (обессоливание воды, циркуляция католита, Au/Ag/МПГ штейн-шлак, закачка шахтных вод) — проверить,
  что структурные фильтры (гео/числа) реально ограничивают выдачу, а не только вектор.
