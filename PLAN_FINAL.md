## Что меняет Yandex AI Studio

До сих пор архитектура ждала «выделения мощностей» под MetalGPT-1 (32B, GPU) и BGE-m3.
Yandex AI Studio снимает это ограничение полностью:

| Слой | Было (план) | Стало (Yandex) |
|---|---|---|
| LLM извлечения/ответов | MetalGPT-1 на GPU (нет железа) | **YandexGPT через API — доступен сейчас** |
| Эмбеддинги | BGE-m3 локально (тяжёлая модель) | **text-search-doc/query API — доступны сейчас** |
| GPU | обязателен | **не нужен вообще** |

Архитектура «интерфейс + реализации» отрабатывает на 100%: добавляем по одной
реализации в `llm_client.py` и `embeddings.py`, всё остальное (extractor, graphrag,
qa, API, UI) не трогаем. История для презентации жюри: *«решение LLM-агностично —
сегодня Yandex AI Studio, в проде Норникеля тот же код работает с MetalGPT-1 on-prem
(клиент уже написан)»*.

## §Y1. LLM-бэкенд `yandex` — `klubok/extraction/llm_client.py`

YandexGPT доступен через **OpenAI-совместимый эндпоинт**, поэтому клиент — вариация
уже написанного `MetalGPTClient`:

- `base_url = https://llm.api.cloud.yandex.net/v1`
- `model = gpt://<FOLDER_ID>/yandexgpt/latest` (Pro; для дешёвого bulk-извлечения
  запасной вариант `yandexgpt-lite/latest` — выбирается конфигом)
- Auth: значение API-ключа как Bearer-токен (OpenAI-протокол), т.е. тот же
  `Authorization: Bearer <API_KEY>` код, что в MetalGPTClient.
- `temperature=0.1` для извлечения (как сейчас), `response_format={"type":"json_object"}`
  если совместимый слой его принимает — пробуем, при ошибке откатываемся на обычный
  режим (лояльный `_loads_lenient` в extractor уже страхует).

Обязательные отличия от MetalGPTClient (выносим в общий базовый класс
`OpenAICompatClient`, чтобы не дублировать):

1. **Ретраи на 429/5xx** с экспоненциальным backoff (3–5 попыток, jitter) — квоты
   Yandex по RPS реально бьют при батч-ингесте.
2. **Троттлинг**: конфигурируемый `yandex_rps` (дефолт 5), простой token-bucket или
   `time.sleep` между вызовами в ingest-скрипте.
3. **Таймаут 180 с** (long-context извлечение с few-shot на Pro бывает медленным).

`get_llm()`: ветка `settings.llm_backend == "yandex"`.

## §Y2. Эмбеддинг-бэкенд `yandex` — `klubok/vectorstore/embeddings.py`

Нативный REST (эмбеддинги в OpenAI-совместимом слое не гарантированы):

- `POST https://llm.api.cloud.yandex.net/foundationModels/v1/textEmbedding`
- Заголовок `Authorization: Api-Key <API_KEY>`
- Тело: `{"modelUri": "emb://<FOLDER_ID>/text-search-doc/latest", "text": "..."}`
- Ответ: `{"embedding": [...], ...}`, **размерность 256**.

Критичные особенности, которых нет у BGE/Mock:

1. **Асимметричные модели**: `text-search-doc` для индексации чанков,
   `text-search-query` для запросов. Текущий Protocol `Embedder.encode(texts)` не
   различает — расширяем контракт: `encode(texts, kind="doc")` + метод
   `encode_query(text)` с дефолтной реализацией через `encode` (Mock/BGE не меняются
   поведенчески). `QdrantStore.search` переводим на `encode_query`.
2. **Один текст на запрос** (нет батча) → для ~5–7 тыс. чанков сабсета нужны:
   - `ThreadPoolExecutor` (4–8 потоков, под RPS-квоту),
   - **кэш эмбеддингов на диске** (`data/cache/emb_cache.sqlite`, ключ — sha1 текста):
     повторные прогоны ingest не платят деньгами и временем. Кэш — отдельный
     маленький модуль `klubok/vectorstore/emb_cache.py`, оборачивающий любой Embedder.
3. **EMBEDDING_DIM=256** при `EMBEDDER_BACKEND=yandex` → Qdrant-коллекция
   пересоздаётся (`ensure_collection(recreate=True)` при смене размерности —
   добавить проверку несоответствия dim коллекции и эмбеддера с понятной ошибкой).

## §Y4. Смоук-тест — новый `scripts/check_yandex.py`

Первое, что запускаем после конфигурации (до любого ингеста):
1. один `complete("Ответь словом: работает")` через YandexLLM → печать ответа/латентности;
2. один `textEmbedding` doc + query → печать размерности;
3. извлечение на одном few-shot тексте из gold-набора → печать распарсенных триплетов.
Выход ненулевой при любом фейле — быстрая диагностика ключа/квот/сети.

## §Y5. Калибровка извлечения на Yandex — `scripts/eval_extraction.py` (готов)

Впервые получаем **реальные метрики** вместо Mock:
1. `LLM_BACKEND=yandex python scripts/eval_extraction.py` (gold, 4 металлургических
   примера) → baseline P/R/F1 по сущностям/рёбрам/атрибутам.
2. Если recall по рёбрам < ~0.6 — итерации: сократить few-shot до 1 примера
  (yandexgpt контекст дорогой), усилить формулировки `_RULES`, попробовать
  `yandexgpt-lite` vs `pro` (скорость/цена vs качество — решение по метрикам).
3. Зафиксировать выбранную конфигурацию в `.env` и в README (для воспроизводимости).

## §Y9. Сквозная проверка по 4 эталонным вопросам ТЗ

На наполненном графе через UI прогнать дословно:
1. обессоливание воды (сульфаты/хлориды 200–300 мг/л, сухой остаток ≤1000 мг/дм³) —
   проверяет `parse_constraint` + числовой Cypher-фильтр;
2. циркуляция католита при электроэкстракции Ni, мировая практика + оптимальная
   скорость потока — проверяет гео-фильтр + синонимы RU/EN;
3. эксперименты/публикации по Au/Ag/МПГ между штейном и шлаком за 5 лет — проверяет
   временной фильтр + Publication/Experiment связки;
4. закачка шахтных вод в РФ и за рубежом + ТЭП — проверяет сравнение
   отечественная/зарубежная.
Каждый прогон — скриншот в `docs/screenshots/` (пойдут в презентацию и README).
